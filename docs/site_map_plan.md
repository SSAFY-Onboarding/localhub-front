현재 MVP 화면은 **7개 라우트**로 구성합니다. 회원가입·로그인은 없으며, 게시글 CRUD는 비밀번호 확인 방식으로 처리합니다. 장소 지도·검색·목록은 하나의 통합 지도 화면에서 제공하고, AI 여행 추천은 별도 페이지에서 제공합니다.

## 1. 전체 정보 구조

```text
LocalHub
├─ 홈
│  ├─ 선정 권역 소개
│  ├─ 지역 지도 바로가기
│  └─ 커뮤니티 바로가기
│
├─ 지역 지도
│  ├─ 7개 콘텐츠 유형 필터
│  ├─ 장소 마커·클러스터
│  └─ 장소 기본정보 팝업
│
├─ AI 여행 추천
│  ├─ 선택형 기준 숙소 검색·선택
│  ├─ 질문 의도·카테고리·지역 판별
│  ├─ 실제 장소 데이터 기반 최대 8곳 추천
│  └─ AI 답변·추천 장소 지도 표시
```

## M-01. 지역 지도 화면

지도 API는 장소명, 콘텐츠 유형, 주소, `latitude`, `longitude`를 경량 응답으로 제공합니다. 프런트는 Leaflet의 현재 화면 경계를 `south`, `west`, `north`, `east`로 전달해 보이는 영역의 장소만 조회합니다.

### 화면 구성

```text
┌────────────────────────────────────────────────────┐
│ 지역 지도                                           │
│ [전체] [관광지] [문화시설] [축제] [여행코스]         │
│ [레포츠] [숙박] [쇼핑]                               │
├────────────────────────────────────────────────────┤
│                                                    │
│                    Leaflet 지도                     │
│         ●     [24]                   ●              │
│                 ●                                  │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 마커 팝업 최소 정보

```text
┌────────────────────────┐
│ 장소명                  │
│ 주소                    │
└────────────────────────┘
```

전화번호와 대표 이미지는 기존 MoSCoW에서 Should로 분류했으므로 Must-only 구현에서는 제외할 수 있습니다.

### 주요 동작

| 사용자 행동    | 결과                  |
| --------- | ------------------- |
| 화면 진입     | 전체 유형의 유효한 장소 마커 표시 |
| 콘텐츠 유형 클릭 | 해당 유형의 선택 상태 토글, 복수 유형 동시 표시 |
| 마커 클릭     | 장소명과 주소 팝업 표시       |
| 숫자 클러스터 클릭 | 해당 영역을 확대해 묶인 마커를 단계적으로 분리 |
| 전체 클릭     | 전체 선택 상태면 모두 해제, 일부·미선택 상태면 모두 선택 |

### 데이터 예외 처리

| 데이터 상태              | 화면 처리                  |
| ------------------- | ---------------------- |
| `latitude` 또는 `longitude` 없음 | 지도 마커에서 제외             |
| 좌표 숫자 변환 실패         | 지도 마커에서 제외             |
| `addr1` 빈 문자열       | `주소 정보 없음` 표시          |
| API 로딩 중            | 지도 중앙에 로딩 표시           |
| API 실패              | `장소 정보를 불러오지 못했습니다` 표시 |
| 표시 가능한 장소 없음        | `표시할 장소가 없습니다` 표시      |

### 지도·목록 통합 탐색

지도 옆 목록에서 현재 조회 영역의 장소를 검색하고 비교합니다. 별도 장소 목록 라우트는 만들지 않으며 지도와 목록은 같은 `MarkerPlace[]` 상태를 공유합니다.

#### 데스크톱

```text
┌──────────────────┬──────────────────────────────────┐
│ [장소명·주소 검색] │                                  │
│ [전체][관광지]...  │                                  │
│ 현재 영역 123개    │              지도                │
│ 장소 목록          │                                  │
│ 장소 목록          │       [이 지역에서 다시 검색]     │
│ [이전] 1/4 [다음]  │                                  │
└──────────────────┴──────────────────────────────────┘
```

- 1024px 초과에서는 약 390px, 761~1024px에서는 약 320px 사이드바를 사용합니다.
- 목록 선택 시 해당 마커로 지도를 이동하고 선택 상태를 강조합니다.
- 지도 사이드바와 마커 팝업에는 장소 썸네일을 표시하며, 이미지가 없거나 실패하면 카테고리별 기본 SVG를 사용합니다.
- 마커 선택 시 마커 위에 썸네일·카테고리·장소명·주소 팝업을 표시하고, 목록의 같은 항목을 강조해 보이는 위치로 스크롤합니다.
- 카테고리 필터는 각 마커와 동일한 색상·SVG 아이콘을 표시해 범례 역할을 겸합니다. 목록 항목도 같은 시각 체계를 사용합니다.
- 지도 줌 14 이하에서는 밀집도를 낮추기 위해 색상 점만 표시하고, 줌 15 이상에서는 확대된 마커 안에 카테고리 아이콘을 표시합니다.
- 현재 검색·카테고리 조건으로 표시된 마커 중 화면상 반경 60px 안에 가까이 모인 항목은 하나의 클러스터로 묶고 내부 장소 수를 숫자로 표시합니다. 클러스터 크기는 2~19개 36px, 20~99개 44px, 100개 이상 52px로 구분합니다.
- 클러스터를 클릭하면 포함된 장소가 분리될 때까지 지도를 확대합니다. 최대 줌에서도 같은 위치에 겹친 마커는 펼침 형태로 분리해 선택할 수 있게 합니다.
- 사이드바에서 클러스터 안의 장소를 선택하면 해당 마커가 보일 때까지 자동으로 확대·이동한 후 팝업을 엽니다.
- 선택 장소 정보를 사이드바 상단에 별도로 삽입하지 않아 기존 목록 위치가 밀리지 않게 합니다.
- 지도 확대·축소가 끝나면 변경된 범위로 자동 재조회합니다. 단순 드래그 이동은 결과를 즉시 바꾸지 않고 `[이 지역에서 다시 검색]` 버튼을 표시합니다.

#### 모바일

목록 사이드바는 지도 하단의 버튼식 바텀 시트로 전환합니다. 드래그 제스처는 MVP에서 제외합니다.

| 단계 | 높이와 내용 |
| --- | --- |
| 축소 | 약 82px, 현재 영역 장소 수와 `[목록 보기]` 표시 |
| 중간 | 화면의 약 48%, 검색·필터와 일부 목록 표시 |
| 전체 | 화면의 약 90%, 검색·필터·전체 목록 탐색 |

시트 상단 버튼을 누를 때 `축소 → 중간 → 전체 → 축소` 순서로 전환합니다. 마커를 누르면 자동으로 중간 상태를 열어 선택한 장소 정보를 보여줍니다.

#### 검색·필터 규칙

- 검색은 현재 `/api/places/map`으로 받은 영역 데이터의 `name`, `address`를 대상으로 합니다.
- 입력할 때마다 검색하지 않고 Enter 또는 `[검색]`을 눌렀을 때 적용합니다.
- 카테고리는 복수 선택하며, 전체 선택 상태에서는 카테고리 없는 API를 한 번 호출합니다.
- 선택 카테고리가 없으면 API를 호출하지 않고 마커와 목록을 비웁니다.
- 검색 또는 카테고리 변경 시 목록 페이지를 1로 초기화합니다.
- 목록은 한 페이지에 12개를 표시합니다.
- 지도 영역 응답이 `truncated: true`이면 확대 안내를 함께 표시합니다.

#### 상태와 접근성

| 상태 | UI 처리 |
| --- | --- |
| 최초·영역 재조회 | 사이드바 목록에 로딩 표시 |
| 검색 결과 없음 | `조건에 맞는 장소가 없습니다`와 검색 초기화 제공 |
| 카테고리 미선택 | `표시할 장소 유형을 선택해 주세요` 표시 |
| API 실패 | 오류 문구와 `[다시 시도]` 제공 |
| 지도 확대·축소 후 | 변경된 지도 범위로 자동 재조회 |
| 지도 드래그 이동 후 | 기존 결과를 유지하고 `[이 지역에서 다시 검색]` 표시 |

검색창과 목록 버튼에는 접근 가능한 이름을 제공하고, 장소 항목은 키보드 포커스와 선택 강조를 지원합니다.

## B-01. 게시글 목록 화면

### 화면 구성

```text
┌──────────────────────────────────────────────────┐
│ 커뮤니티                              [글쓰기]    │
├──────┬──────────────────────┬──────┬──────┬────────┤
│ 번호 │ 제목                 │ 조회 │ 추천 │ 작성일  │
├──────┼──────────────────────┼──────┼──────┼────────┤
│ 10   │ 여행 코스 추천        │ 128  │ 15   │ 07/14  │
│  9   │ 문화시설 방문 후기    │  37  │  4   │ 07/14  │
│  8   │ 숙소 주변 관광지 추천 │  21  │  2   │ 07/13  │
└──────┴──────────────────────┴──────┴──────┴────────┘
│               [이전] 1 2 3 [다음]                │
└──────────────────────────────────────────────────┘
```

### 주요 동작

| 사용자 행동    | 결과               |
| --------- | ---------------- |
| 화면 진입     | 최신순 게시글 목록 조회    |
| 게시글 제목 클릭 | `/posts/:id`로 이동 |
| 글쓰기 클릭    | `/posts/new`로 이동 |
| 페이지 번호 클릭 | `/posts?page={page}`로 이동 후 해당 페이지 조회 |

페이지 상태는 URL 쿼리스트링에 보존합니다. 상세 화면에서 목록으로 돌아올 때도 이전 `page` 값을 유지하며, API 요청의 `size`는 MVP에서 `10`으로 고정합니다. 유효하지 않은 페이지 값은 `1`로 보정합니다.

게시글 API는 향후 확장을 위해 `keyword` 검색을 지원하지만, 검색은 Should 범위이므로 MVP 목록 화면에는 검색창을 노출하지 않습니다. 현재 챗봇도 커뮤니티 게시글 검색은 지원하지 않습니다.

### 화면 상태

* 로딩: 목록 대신 스켈레톤 또는 `불러오는 중`
* 빈 목록: `아직 등록된 게시글이 없습니다`와 `[글쓰기]` 버튼 표시
* 페이지 범위 초과: 빈 목록을 표시하고 첫 페이지로 이동할 수 있는 버튼 제공
* 오류: 오류 메시지와 재시도 버튼 표시

데스크톱에서는 번호·제목·조회수·추천 수·작성일 테이블을 사용합니다. 모바일에서는 중요도가 낮은 번호를 숨기고 제목과 집계·작성일을 읽기 쉬운 형태로 재배치합니다.

### 화면 구성

```text
┌─────────────────────────────────────────────────┐

│ 작성일: 2026.07.14 · 조회 129                    │

│                                                 │
```

### 주요 동작

| 수정 클릭        | `/posts/:id/edit`로 바로 이동 |
| 삭제 클릭        | 삭제용 비밀번호 확인 모달 표시 |
| 존재하지 않는 글 진입 | 404 안내 후 목록 버튼 표시 |
| 일정 지도 마커 클릭 | Day·순서·장소명·시간·메모 팝업 표시 |
| 추천 클릭 | 인증 없이 추천 수 1 증가, 처리 중 버튼 비활성화 |

여행 일정에 좌표가 있는 장소가 하나 이상 있을 때만 일정 지도를 표시합니다. 날짜별로 마커와 연결선 색을 구분하며 좌표가 없는 장소는 타임라인에는 유지하고 지도에서만 제외합니다.

현재 백엔드의 상세 조회 API는 호출할 때마다 조회수를 1 증가시킵니다. 따라서 상세 화면 진입뿐 아니라 수정 화면에서 기존 데이터를 불러오는 요청도 조회수에 포함됩니다.

### 삭제용 비밀번호 확인 모달

```text
┌────────────────────────────┐
│ 게시글 삭제                  │
│ [비밀번호 입력           ]  │
│ 비밀번호가 일치하면 게시글이 │
│ 즉시 삭제됩니다.             │
│                            │
│ [취소]             [삭제]  │
└────────────────────────────┘
```

#### 삭제 흐름

```text
삭제 클릭
→ 비밀번호 입력
→ 서버 검증 및 삭제
→ 성공: 목록 화면 이동
→ 비밀번호 불일치: 모달 유지, 비밀번호만 비우고 오류 메시지 표시
```

삭제 요청 중에는 취소·삭제 버튼을 비활성화하여 중복 요청을 방지합니다. 별도의 삭제 확인창은 추가하지 않습니다.

## B-04. 게시글 수정 화면

게시글 작성 화면과 동일한 `PostForm` 컴포넌트를 사용합니다.

### 차이점

| 항목      | 작성 화면    | 수정 화면            |
| ------- | -------- | ---------------- |
| 제목      | 빈 값      | 기존 제목            |
| 내용      | 빈 값      | 기존 내용            |
| 비밀번호    | 폼에서 신규 입력 | 수정 제출 시 모달에서 입력 |
| 실행 버튼   | 등록       | 수정 완료            |
| 성공 후 이동 | 새 게시글 상세 | 기존 게시글 상세        |
| 실패 시 | 입력값 유지 | 수정 내용 유지 |

수정 화면에 직접 접근할 수 있으므로 진입 시 `GET /api/posts/{id}`로 기존 제목과 내용을 조회합니다. 조회 중에는 폼을 비활성화하고, 404이면 게시글 목록 이동 버튼을, 서버 오류이면 재시도 버튼을 표시합니다.

```text
수정 화면 진입
→ 기존 게시글 조회
→ 제목·내용 수정
→ 수정 완료 클릭
→ 비밀번호 모달 표시
→ 비밀번호와 수정 내용을 함께 서버로 전송
→ 성공: 게시글 상세로 이동
```

별도 비밀번호 검증 API는 사용하지 않습니다. `403 INVALID_PASSWORD`이면 수정 내용과 모달을 유지하고 비밀번호 입력값만 비운 뒤 해당 필드에 포커스를 이동합니다. 요청 중에는 확인 버튼을 비활성화합니다. 비밀번호는 URL, 라우터 상태, Pinia, `localStorage`에 저장하지 않습니다. 취소하면 해당 게시글 상세 화면으로 돌아갑니다.

## C-01. AI 여행 추천

AI 여행 추천은 `/chatbot`에서 제공하는 전용 화면입니다. 사용자는 숙소를 선택해 기준점으로 쓰거나, 숙소 없이 질문에 서울 지역을 포함할 수 있습니다. 서버는 질문의 의도·카테고리·지역을 판별하고 실제 장소 데이터에서 최대 8곳을 골라, 해당 데이터에 근거한 AI 답변과 지도 정보를 반환합니다. 상세 API 계약과 현재 제약은 [`챗봇_MVP_기능_명세.md`](./챗봇_MVP_기능_명세.md)를 따릅니다.

### 데스크톱

```text
┌──────────────────────────────┬──────────────────────────────┐
│ AI 여행 추천                 │ 추천 장소 지도                │
│ [(선택) 기준 숙소 선택/변경]  │ 기준점 또는 추천 장소 범위      │
│                              │ 최대 8개 추천 장소 마커         │
│ 챗봇 답변과 사용자 메시지     │ 목록 선택 시 마커 포커스        │
│ 추천 장소 최대 8곳 목록       │                              │
│ [질문 입력]          [전송]  │                              │
└──────────────────────────────┴──────────────────────────────┘
```

첫 질문 전에는 채팅 패널을 중심으로 표시합니다. 추천 장소가 포함된 응답을 받으면 채팅 패널과 지도 패널을 나란히 표시합니다.

### 모바일

```text
┌──────────────────────────┐
│ AI 여행 추천             │
│ [(선택) 기준 숙소 선택]   │
│ 대화 및 추천 장소 목록    │
│ [질문 입력]      [전송]  │
├──────────────────────────┤
│ 추천 장소 지도            │
└──────────────────────────┘
```

820px 이하에서는 채팅 다음에 지도를 세로로 배치합니다.

### 지원 기능과 상태

* 숙소 선택은 선택 사항입니다. 선택 모달은 `GET /api/places`를 사용해 서울의 `숙박` 장소를 검색하며, 6개 단위 페이지네이션을 제공합니다.
* 첫 질문 전에 `서울역 근처 축제`, `성수의 전시와 쇼핑`, `한강 주변 체험 장소` 빠른 질문을 제공합니다.
* 질문을 보내면 사용자 메시지를 추가하고, 응답 중에는 입력창과 전송 버튼을 비활성화하며 로딩 애니메이션을 표시합니다.
* 이전 대화는 `history`로 전송하며, 서버는 최근 메시지를 AI 답변 생성 문맥으로 사용합니다.
* 응답은 AI 답변, 추천 장소 최대 8곳, 지도 기준점, 서버가 판별한 질의 유형을 포함합니다.
* 추천 장소 목록을 선택하면 해당 지도 마커로 이동해 팝업을 열고, 지도 마커를 선택하면 목록의 같은 장소를 활성화합니다.
* 숙소를 변경하면 추천 장소와 선택 상태는 초기화하지만 화면에 표시된 이전 메시지는 유지합니다.
* API 요청 실패 시 오류 문구와 `다시 시도` 버튼을 표시합니다.

### 현재 MVP 제약

* 추천은 키워드 일치도와 기준점 직선거리를 사용하므로 실제 도로 이동 거리·시간·교통수단을 반영하지 않습니다.
* 질문 분류용 AI 호출이 실패하면 규칙 기반 분류기로 대체하지만, 답변 생성용 API를 사용할 수 없으면 장소 목록과 함께 안내 문구를 반환합니다.
* 커뮤니티 게시글 검색, 영업시간·가격·혼잡도 등 실시간 정보 보장, 실제 경로 최적화는 지원하지 않습니다.
* 대화와 선택한 숙소는 브라우저 저장소에 보존하지 않으므로 새로고침하거나 페이지를 벗어나면 초기화됩니다.
* 회원 식별, 대화 저장, 스트리밍 응답은 지원하지 않습니다.

# 6. 화면과 API 연결

| 화면     | API                                  | 사용 목적        |
| ------ | ------------------------------------ | ------------ |
| 지역 지도  | `GET /api/places/map?south=&west=&north=&east=` | 현재 지도 영역의 경량 마커 조회 |
| 지역 지도  | `GET /api/places/map?...&category=관광지` | 유형별 지도 마커 조회 |
| 지역 지도 | `GET /api/places/categories` | 콘텐츠 유형 필터 구성 |
| 지역 지도 | `GET /api/places/{place_id}` | `/map?place=` 진입 시 선택 장소 위치 확인 |
| 게시글 목록 | `GET /api/posts?page=1&size=10`      | 게시글 목록       |
| 게시글 상세 | `GET /api/posts/{id}`                | 게시글 상세 조회 및 조회수 증가 |

| 게시글 수정 | `PUT /api/posts/{id}`                | 비밀번호 확인 및 수정 |

| AI 여행 추천 | `GET /api/places?category=숙박&region=서울` | 기준 숙소 검색·선택 |
| AI 여행 추천 | `POST /api/chat`                     | 숙소 또는 질문 지역 기준 장소 추천 |

## 최종 화면 범위

현재 MVP 기준으로 구현하는 화면은 다음과 같습니다.

> **홈 1개 + 지도·목록 통합 화면 1개 + AI 여행 추천 1개 + 게시판 4개**

장소 지도·목록·검색은 `/map`에 통합합니다. 이후 고급 정렬·상세 필터·검색엔진 노출이 필요해지면 `/places`와 `/places/:place_id`를 분리할 수 있지만, MVP에서는 지도와 목록의 양방향 연동 및 모바일 바텀 시트를 우선합니다. 회원·관리자 화면은 현재 범위에서 제외합니다.
