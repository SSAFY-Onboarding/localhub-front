Must-have만 구현한다면 화면을 **6개 라우트 + 전역 챗봇 위젯 1개**로 제한하는 것이 가장 안전합니다. 회원가입·로그인은 없으며, 게시글 CRUD는 비밀번호 확인 방식으로 처리합니다. 지도는 선택 기능 최소 1개 조건을 충족하기 위한 MVP 필수 화면으로 포함합니다. 

## 1. 전체 정보 구조

```text
LocalHub
├─ 홈
│  ├─ 선정 권역 소개
│  ├─ 지역 지도 바로가기
│  └─ 커뮤니티 바로가기
│
├─ 지역 지도
│  ├─ 7개 콘텐츠 유형 필터
│  ├─ 장소 마커
│  └─ 장소 기본정보 팝업
│
├─ 커뮤니티
│  ├─ 게시글 목록
│  ├─ 게시글 상세
│  ├─ 게시글 작성
│  └─ 게시글 수정
│
└─ 챗봇 위젯
   ├─ 지역 정보 질문
   ├─ 커뮤니티 게시글 검색 질문
   └─ 대화 히스토리
```

## 2. 라우트 구조

| 화면 ID | 화면명    | 경로                | 핵심 목적                |
| ----- | ------ | ----------------- | -------------------- |
| H-01  | 홈      | `/`               | 서비스 소개와 지도·커뮤니티 진입   |
| M-01  | 지역 지도  | `/map`            | 제공 관광 데이터의 위치 확인     |
| B-01  | 게시글 목록 | `/posts`          | 익명 게시글 목록 조회         |
| B-02  | 게시글 상세 | `/posts/:id`      | 게시글 내용 조회 및 수정·삭제 진입 |
| B-03  | 게시글 작성 | `/posts/new`      | 새 게시글 등록             |
| B-04  | 게시글 수정 | `/posts/:id/edit` | 기존 내용을 조회하고 비밀번호 확인 후 수정 |
| C-01  | 챗봇 위젯  | 전 화면 공통           | 지역 정보 및 게시글 자연어 질의   |

작성 화면과 수정 화면은 라우트는 나누되, 실제 구현에서는 하나의 `PostForm` 컴포넌트를 재사용합니다.

```text
┌──────────────────────────────────────────────┐

├──────────────────────────────────────────────┤
│ 서비스 안내                                  │
│ · 7개 유형 관광 정보 제공                     │
│ · 익명 게시판 운영                            │
│ · 지역 정보 챗봇 제공                         │

```

### 주요 동작

| ----------- | ------------ |

최근 게시글 목록이나 관광지 추천 카드는 Must 범위에서 제외하여 홈을 단순하게 유지합니다.

## M-01. 지역 지도 화면

제공 데이터에는 장소명, 콘텐츠 유형, 주소, 경도·위도가 공통으로 포함됩니다. `mapx`, `mapy`는 문자열이므로 백엔드 또는 프런트에서 숫자로 변환해야 합니다. 

### 화면 구성

```text
┌────────────────────────────────────────────────────┐
│ 지역 지도                                           │
│ [전체] [관광지] [문화시설] [축제] [여행코스]         │
│ [레포츠] [숙박] [쇼핑]                               │
├────────────────────────────────────────────────────┤
│                                                    │
│                    Leaflet 지도                     │
│         ●      ●                    ●              │
│                 ●                                  │
│                                                    │
└────────────────────────────────────────────────────┘
```

### 마커 팝업 최소 정보

```text
┌────────────────────────┐
│ 장소명                  │
│ 주소                    │
└────────────────────────┘
```

전화번호와 대표 이미지는 기존 MoSCoW에서 Should로 분류했으므로 Must-only 구현에서는 제외할 수 있습니다.

### 주요 동작

| 사용자 행동    | 결과                  |
| --------- | ------------------- |
| 화면 진입     | 전체 유형의 유효한 장소 마커 표시 |
| 콘텐츠 유형 클릭 | 선택한 유형의 마커만 표시      |
| 마커 클릭     | 장소명과 주소 팝업 표시       |
| 전체 클릭     | 모든 유형 마커 다시 표시      |

### 데이터 예외 처리

| 데이터 상태              | 화면 처리                  |
| ------------------- | ---------------------- |
| `mapx` 또는 `mapy` 없음 | 지도 마커에서 제외             |
| 좌표 숫자 변환 실패         | 지도 마커에서 제외             |
| `addr1` 빈 문자열       | `주소 정보 없음` 표시          |
| API 로딩 중            | 지도 중앙에 로딩 표시           |
| API 실패              | `장소 정보를 불러오지 못했습니다` 표시 |
| 표시 가능한 장소 없음        | `표시할 장소가 없습니다` 표시      |

## B-01. 게시글 목록 화면

### 화면 구성

```text
┌──────────────────────────────────────────────────┐
│ 커뮤니티                              [글쓰기]    │
├──────┬───────────────────────────────┬───────────┤
│ 번호 │ 제목                          │ 작성일     │
├──────┼───────────────────────────────┼───────────┤
│ 10   │ 여행 코스 추천 부탁드립니다    │ 07/14     │
│  9   │ 문화시설 방문 후기             │ 07/14     │
│  8   │ 숙소 주변 관광지 추천          │ 07/13     │
└──────┴───────────────────────────────┴───────────┘
│               [이전] 1 2 3 [다음]                │
└──────────────────────────────────────────────────┘
```

### 주요 동작

| 사용자 행동    | 결과               |
| --------- | ---------------- |
| 화면 진입     | 최신순 게시글 목록 조회    |
| 게시글 제목 클릭 | `/posts/:id`로 이동 |
| 글쓰기 클릭    | `/posts/new`로 이동 |
| 페이지 번호 클릭 | `/posts?page={page}`로 이동 후 해당 페이지 조회 |

페이지 상태는 URL 쿼리스트링에 보존합니다. 상세 화면에서 목록으로 돌아올 때도 이전 `page` 값을 유지하며, API 요청의 `size`는 MVP에서 `10`으로 고정합니다. 유효하지 않은 페이지 값은 `1`로 보정합니다.

게시글 API는 향후 확장을 위해 `keyword` 검색을 지원하지만, 검색은 Should 범위이므로 MVP 목록 화면에는 검색창을 노출하지 않습니다. 챗봇의 게시글 검색 질의는 별도로 지원합니다.

### 화면 상태

* 로딩: 목록 대신 스켈레톤 또는 `불러오는 중`
* 빈 목록: `아직 등록된 게시글이 없습니다`와 `[글쓰기]` 버튼 표시
* 페이지 범위 초과: 빈 목록을 표시하고 첫 페이지로 이동할 수 있는 버튼 제공
* 오류: 오류 메시지와 재시도 버튼 표시

데스크톱에서는 번호·제목·작성일 테이블을 사용합니다. 모바일에서는 번호를 숨기고 제목과 작성일을 세로로 배치한 카드형 목록으로 전환합니다.

삭제 클릭

삭제 요청 중에는 취소·삭제 버튼을 비활성화하여 중복 요청을 방지합니다. 별도의 삭제 확인창은 추가하지 않습니다.

## B-03. 게시글 작성 화면

### 화면 구성

```text
┌─────────────────────────────────────────────────┐
│ 게시글 작성                                      │
├─────────────────────────────────────────────────┤
│ 제목                                             │
│ [                                             ]  │
│                                                 │
│ 내용                                             │
│ [                                             ]  │
│ [                                             ]  │
│                                                 │
│ 수정용 비밀번호                                  │
│ [                    ]                           │
│ 게시글 수정·삭제 시 필요합니다.                   │
│                                                 │
│                           [취소] [등록]           │
└─────────────────────────────────────────────────┘
```

### 입력 검증

| 필드       | 검증            |
| -------- | ------------- |
| 제목       | 앞뒤 공백 제거 후 1~100자 |
| 내용       | 앞뒤 공백 제거 후 1~5000자 |
| 수정용 비밀번호 | 4~20자, 공백만 입력 불가 |

제목과 내용은 프런트에서도 공백을 정규화하지만, 비밀번호는 앞뒤 공백을 자동 제거하지 않습니다. 백엔드 검증을 최종 기준으로 하며 `422` 응답의 필드별 오류를 해당 입력란 아래에 표시합니다.

### 사용자 흐름

```text
게시글 목록
→ 글쓰기
→ 제목·내용·비밀번호 입력
→ 등록
→ 성공: 작성된 게시글 상세 화면
→ 실패: 입력값 유지 + 오류 메시지
```

등록 요청 중에는 등록 버튼을 비활성화합니다. 취소하면 게시글 목록으로 이동합니다.

## B-04. 게시글 수정 화면

게시글 작성 화면과 동일한 `PostForm` 컴포넌트를 사용합니다.

### 차이점

| 항목      | 작성 화면    | 수정 화면            |
| ------- | -------- | ---------------- |
| 제목      | 빈 값      | 기존 제목            |
| 내용      | 빈 값      | 기존 내용            |
| 비밀번호    | 폼에서 신규 입력 | 수정 제출 시 모달에서 입력 |
| 실행 버튼   | 등록       | 수정 완료            |
| 성공 후 이동 | 새 게시글 상세 | 기존 게시글 상세        |
| 실패 시 | 입력값 유지 | 수정 내용 유지 |

수정 화면에 직접 접근할 수 있으므로 진입 시 `GET /api/posts/{id}`로 기존 제목과 내용을 조회합니다. 조회 중에는 폼을 비활성화하고, 404이면 게시글 목록 이동 버튼을, 서버 오류이면 재시도 버튼을 표시합니다.

```text
수정 화면 진입
→ 기존 게시글 조회
→ 제목·내용 수정
→ 수정 완료 클릭
→ 비밀번호 모달 표시
→ 비밀번호와 수정 내용을 함께 서버로 전송
→ 성공: 게시글 상세로 이동
```

별도 비밀번호 검증 API는 사용하지 않습니다. `403 INVALID_PASSWORD`이면 수정 내용과 모달을 유지하고 비밀번호 입력값만 비운 뒤 해당 필드에 포커스를 이동합니다. 요청 중에는 확인 버튼을 비활성화합니다. 비밀번호는 URL, 라우터 상태, Pinia, `localStorage`에 저장하지 않습니다. 취소하면 해당 게시글 상세 화면으로 돌아갑니다.

## C-01. 챗봇 위젯

챗봇은 특정 페이지가 아니라 모든 화면에 표시되는 전역 컴포넌트입니다. RFP는 제공 JSON 기반 질의응답, 게시글 검색 등의 주요 질의 대응과 플로팅·대화 히스토리·모바일 대응 UI를 요구합니다. 

### 접힌 상태

```text
화면 우측 하단
        ┌──────┐
        │ 챗봇 │
        └──────┘
```

### 데스크톱 펼친 상태

```text
┌─────────────────────────────┐
│ LocalHub 챗봇          [×]  │
├─────────────────────────────┤
│ 안녕하세요. 지역 정보를     │
│ 물어보세요.                  │
│                             │
│          관광지 추천해 줘    │
│                             │
│ 추천 장소는 다음과 같습니다 │
├─────────────────────────────┤
│ [메시지를 입력하세요] [전송] │
└─────────────────────────────┘
```

### 모바일 펼친 상태

```text
┌──────────────────────────┐
│ LocalHub 챗봇       [×]  │
├──────────────────────────┤
│                          │
│       전체 대화 영역      │
│                          │
├──────────────────────────┤
│ [메시지 입력]     [전송] │
└──────────────────────────┘
```

### 지원 질문 범위

| 질문 유형  | 예시               | 처리 데이터      |
| ------ | ---------------- | ----------- |
| 관광 정보  | `추천 관광지를 알려줘`    | 관광지 JSON    |
| 문화시설   | `주변 문화시설이 있어?`   | 문화시설 JSON   |
| 숙박     | `숙박 시설을 알려줘`     | 숙박 JSON     |
| 레포츠    | `레포츠 장소 추천해 줘`   | 레포츠 JSON    |
| 쇼핑     | `쇼핑할 곳을 알려줘`     | 쇼핑 JSON     |
| 여행코스   | `여행 코스를 알려줘`     | 여행코스 JSON   |
| 축제 정보  | `축제 장소를 알려줘`     | 축제공연행사 JSON |
| 게시글 검색 | `숙박에 관한 게시글 찾아줘` | SQLite 게시글  |

축제 데이터에는 실제 행사 시작일과 종료일이 없으므로, 챗봇이 축제 일정을 묻는 질문에는 날짜를 임의 생성하지 않습니다.

```text
제공 데이터에 행사 날짜 정보가 없어 정확한 일정을 안내할 수 없습니다.
축제 장소 정보는 안내할 수 있습니다.
```

### 대화 상태

* 대화 히스토리는 현재 브라우저 세션 동안 유지
* 페이지 이동 후에도 유지하려면 Pinia 또는 상위 `App.vue` 상태 사용
* 새로고침 후 보존은 Must 범위에서 제외
* API 요청 중 입력창과 전송 버튼 비활성화
* 답변 실패 시 `답변을 불러오지 못했습니다. 다시 시도해 주세요.` 표시

# 6. 화면과 API 연결

| 화면     | API                                  | 사용 목적        |
| ------ | ------------------------------------ | ------------ |
| 지역 지도  | `GET /api/locations`                 | 전체 장소 조회     |
| 지역 지도  | `GET /api/locations?content_type=12` | 유형별 장소 조회    |
| 게시글 목록 | `GET /api/posts?page=1&size=10`      | 게시글 목록       |
| 게시글 상세 | `GET /api/posts/{id}`                | 게시글 상세       |
| 게시글 작성 | `POST /api/posts`                    | 게시글 등록       |
| 게시글 수정 | `PUT /api/posts/{id}`                | 비밀번호 확인 및 수정 |
| 게시글 삭제 | `DELETE /api/posts/{id}`             | 비밀번호 확인 및 삭제 |
| 챗봇     | `POST /api/chat`                     | 지역·게시글 질의응답  |

수정·삭제 요청 예시는 다음처럼 통일하는 것이 단순합니다.

```json
{
  "password": "1234"
}
```

수정 요청은 제목과 내용을 함께 전달합니다.

```json
{
  "title": "수정된 제목",
  "content": "수정된 내용",
  "password": "1234"
}
```

# 7. 공통 화면 상태 규칙

| 상태       | UI 처리                        |
| -------- | ---------------------------- |
| 로딩       | 콘텐츠 영역에 로딩 표시하고 관련 입력·실행 버튼 비활성화 |
| 게시글 목록 없음 | 안내 문구와 `[글쓰기]` 버튼 표시 |
| 지도 조회 결과 없음 | `표시할 장소가 없습니다` 표시 |
| 입력 오류    | 해당 필드 아래 오류 문구, 입력값 유지 |
| 서버 오류    | 화면 상단 또는 토스트로 오류 안내하고 재시도 제공 |
| 게시글 404 | `요청한 게시글을 찾을 수 없습니다`와 목록 버튼 |
| 삭제 성공    | 게시글 목록으로 이동 후 완료 메시지         |
| 작성·수정 성공 | 해당 게시글 상세 화면으로 이동            |
| 변경 요청 중 | 실행 버튼 비활성화로 중복 요청 방지 |
| 챗봇 요청 중  | 전송 버튼 비활성화 및 응답 대기 표시        |

## 최종 화면 범위

Must-have 기준으로 실제 구현해야 하는 화면은 다음과 같습니다.

> **홈 1개 + 지도 1개 + 게시판 4개 + 전역 챗봇 1개**

이 구조에서는 별도 장소 목록·장소 상세·회원·관리자 화면을 만들지 않습니다. 관광 데이터는 지도와 챗봇에서만 사용하고, 커뮤니티는 CRUD에만 집중하여 3일 일정에서 화면 수와 API 연동 복잡도를 최소화합니다.
