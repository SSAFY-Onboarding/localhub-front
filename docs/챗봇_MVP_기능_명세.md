# AI 여행 추천 챗봇 MVP 기능 명세

> 상태: 프런트·백엔드 합의용 초안  
> 기준 화면: `/chatbot`  
> 목표: 사용자가 기준 숙소를 선택하고 질문하면 실제 장소 데이터에 기반한 주변 장소를 추천받아 지도에서 확인할 수 있게 한다.

## 1. MVP 완료 기준

다음 사용자 흐름이 실제 API와 데이터로 처음부터 끝까지 동작하면 챗봇 MVP가 완료된 것으로 본다.

```text
숙소 검색·선택
→ 여행 관련 질문 입력
→ 답변과 추천 장소 수신
→ 추천 이유·거리 확인
→ 추천 장소를 지도에서 확인
→ 같은 숙소를 기준으로 후속 질문
```

정적인 예시 답변, 하드코딩된 숙소와 추천 장소, 동작하지 않는 추천 버튼은 완료 상태로 보지 않는다.

## 2. Must-have 기능

### 2.1 실제 숙소 선택

- 숙소 목록은 하드코딩하지 않고 장소 API에서 `숙박` 카테고리로 조회한다.
- 숙소명과 주소로 검색할 수 있다.
- 선택 결과에는 최소 `id`, `name`, `address`, `latitude`, `longitude`, `image_url`이 있어야 한다.
- 숙소를 선택하기 전에는 숙소 기반 추천 질문을 전송할 수 없다.
- 선택한 숙소를 변경할 수 있다.
- 대화 시작 후 숙소를 변경하면 기존 대화와 추천 결과를 초기화하고 새로운 대화 세션을 시작한다.

### 2.2 질문 전송과 응답 상태

- 사용자는 자유 입력 또는 빠른 질문 버튼으로 메시지를 보낼 수 있다.
- 공백만 있는 메시지는 전송하지 않는다.
- 메시지는 최대 500자로 제한한다.
- 한 번에 하나의 요청만 보내며 응답 중에는 중복 전송을 막는다.
- 요청 중에는 `답변을 작성하고 있습니다` 상태를 표시한다.
- 요청 실패 시 오류 메시지와 `[다시 시도]`를 제공한다.
- 사용자는 `[새 대화]`로 현재 대화와 추천 결과를 초기화할 수 있다.

### 2.3 실제 장소 데이터 기반 답변

- 챗봇이 추천하는 장소는 장소 DB에 존재하는 `id`를 가져야 한다.
- 모델이 DB에 없는 장소명, 주소, 좌표, 거리, 날짜를 임의로 만들지 않게 한다.
- 추천 장소에서 기준 숙소 자체는 제외한다.
- 한 답변에서 같은 `id`를 중복 추천하지 않는다.
- 한 번에 표시하는 추천 장소는 최대 5개로 제한한다.
- 장소 데이터에 없는 영업시간이나 축제 날짜는 추정하지 않고 정보가 없다고 안내한다.
- 거리는 백엔드가 숙소와 장소 좌표로 계산하며, MVP에서는 `직선거리`임을 화면에 표시한다.

### 2.4 구조화된 응답

챗봇 API는 화면 표시용 문장과 추천 장소를 분리해서 반환한다. 추천 장소를 답변 문자열 안에만 포함하면 안 된다.

```json
{
  "answer": "숙소에서 가까운 문화시설 3곳을 추천했어요.",
  "places": [
    {
      "id": "14-123456",
      "name": "서울시립미술관",
      "category": "문화시설",
      "address": "서울특별시 중구 덕수궁길 61",
      "latitude": 37.5641,
      "longitude": 126.9738,
      "image_url": null,
      "distance_km": 1.2,
      "reason": "숙소에서 가깝고 실내 관람이 가능한 장소입니다."
    }
  ],
  "suggested_questions": [
    "도보 20분 이내만 보여줘",
    "실내 장소로 다시 추천해줘"
  ]
}
```

각 추천 결과는 해당 어시스턴트 메시지에 귀속한다. 새 질문을 보냈다는 이유로 이전 답변의 추천 장소가 다른 결과로 바뀌면 안 된다.

### 2.5 후속 질문 맥락

- MVP는 서버에 세션을 저장하지 않는 무상태 방식으로 구현한다.
- 프런트는 후속 요청에 현재 대화의 최근 메시지를 `history`로 함께 보낸다.
- `history`는 오래된 순서로 최대 10개 메시지만 전달한다.
- 다음과 같은 후속 질문을 처리할 수 있어야 한다.
  - `그중에서 실내 장소만 알려줘`
  - `두 번째 장소 대신 다른 곳 추천해줘`
  - `도보로 가까운 순서로 보여줘`
- 새 대화 또는 숙소 변경 시 프런트의 `history`와 추천 결과를 비운다.
- 브라우저 새로고침 이후 대화 복구는 MVP 범위에 포함하지 않는다.

### 2.6 추천 장소와 지도 연동

- 추천 결과가 있을 때 실제 지도와 마커를 표시한다.
- 기준 숙소 마커와 추천 장소 마커를 시각적으로 구분한다.
- 추천 카드와 지도 마커는 같은 장소 `id`로 연결한다.
- 추천 카드를 선택하면 해당 마커로 지도를 이동하고 팝업을 연다.
- 지도 마커를 선택하면 대응하는 추천 카드를 강조한다.
- 추천 결과가 없으면 빈 지도를 보여주지 않고 결과 없음 안내를 표시한다.
- 모바일에서는 `[대화] / [지도]` 전환 또는 지도 바텀시트 중 하나를 사용해 긴 대화 아래에 지도가 묻히지 않게 한다.

## 3. 백엔드 API 명세

### 3.1 필요한 엔드포인트

| 기능 | 메서드 | 엔드포인트 | 구현 상태 | 조치 |
|---|---|---|---|---|
| 숙소 검색 | `GET` | `/api/places` | 구현됨 | 기존 API 재사용 |
| 챗봇 질문 | `POST` | `/api/chat` | 구현됨 | 요청·응답 스키마와 내부 로직 수정 |

별도의 추천 장소 상세 API와 지도 API는 추가하지 않는다. 챗봇 응답이 지도 표시에 필요한 장소 요약과 좌표를 포함하므로 기존 장소 상세 조회는 사용자가 별도 상세 정보를 요청할 때만 사용한다.

### 3.2 숙소 검색

기존 장소 목록 API를 다음과 같이 호출한다.

```http
GET /api/places?category=숙박&region=서울&keyword=신라&page=1&size=10
```

응답은 기존 `PlaceListResponse`를 그대로 사용한다. 프런트는 `items`를 숙소 선택 모달에 표시하며 좌표가 없는 숙소는 선택 불가 처리하거나 검색 결과에서 제외한다.

```json
{
  "items": [
    {
      "id": "32-142001",
      "name": "서울신라호텔",
      "category": "숙박",
      "address": "서울특별시 중구 동호로 249",
      "latitude": 37.5565,
      "longitude": 127.0052,
      "image_url": null,
      "description": null,
      "phone": null
    }
  ],
  "page": 1,
  "size": 10,
  "total": 1,
  "total_pages": 1
}
```

### 3.3 챗봇 요청

```http
POST /api/chat
Content-Type: application/json
```

```json
{
  "accommodation_id": "32-142001",
  "message": "숙소 주변 문화시설 추천해줘",
  "history": [
    {
      "role": "user",
      "content": "가까운 장소를 추천해줘"
    },
    {
      "role": "assistant",
      "content": "숙소에서 가까운 장소 5곳을 추천했어요.",
      "place_ids": ["12-126508", "14-129897"]
    }
  ]
}
```

| 필드 | 타입 | 필수 | 규칙 |
|---|---|:---:|---|
| `accommodation_id` | string | O | 장소 API에 존재하는 `숙박` 카테고리 ID |
| `message` | string | O | 공백 제거 후 1~500자 |
| `history` | array | X | 오래된 순서, 최대 10개, 기본값 `[]` |

`history` 항목은 다음 형식만 허용한다.

| 필드 | 타입 | 규칙 |
|---|---|---|
| `role` | string | `user` 또는 `assistant` |
| `content` | string | 공백 제거 후 1~3000자 |
| `place_ids` | array | 해당 답변에서 추천했던 장소 ID, 최대 5개, 기본값 `[]` |

현재 질문은 `history`에 중복해서 넣지 않는다. 시스템 프롬프트는 서버가 관리하며 프런트가 `system` 역할 메시지를 보낼 수 없게 한다.

### 3.4 챗봇 성공 응답

**200 OK**

```json
{
  "answer": "서울신라호텔에서 가까운 문화시설을 추천했어요.",
  "places": [
    {
      "id": "14-129897",
      "name": "국립극장",
      "category": "문화시설",
      "address": "서울특별시 중구 장충단로 59",
      "latitude": 37.5529,
      "longitude": 127.0001,
      "image_url": null,
      "distance_km": 0.63,
      "reason": "숙소에서 직선거리 0.63km에 있는 문화시설입니다."
    }
  ],
  "suggested_questions": [
    "도보로 더 가까운 곳만 보여줘",
    "관광지도 추천해줘"
  ]
}
```

| 필드 | 타입 | 규칙 |
|---|---|---|
| `answer` | string | 사용자에게 표시할 답변, 빈 문자열 불가 |
| `places` | array | 0~5개의 구조화된 추천 장소 |
| `suggested_questions` | array | 0~3개의 후속 질문 문구 |

`places`의 `id`, 이름, 카테고리, 주소, 좌표, 이미지는 장소 서비스에서 가져온다. LLM 출력으로 이 필드들을 생성하거나 덮어쓰지 않는다. `distance_km`와 `reason`은 백엔드가 계산 결과로 생성한다.

추천할 장소가 없어도 요청 자체가 성공했다면 `200`과 빈 `places`를 반환하고, `answer`에서 조건에 맞는 장소가 없음을 안내한다.

### 3.5 오류 응답

기존 공통 오류 형식을 사용한다.

```json
{
  "error": {
    "code": "ACCOMMODATION_NOT_FOUND",
    "message": "기준 숙소를 찾을 수 없습니다.",
    "status": 404,
    "details": []
  }
}
```

| code | HTTP | 발생 조건 |
|---|---:|---|
| `VALIDATION_ERROR` | 422 | ID 형식, 메시지 길이, history 형식 위반 |
| `ACCOMMODATION_NOT_FOUND` | 404 | 장소 ID가 없거나 `숙박` 카테고리가 아님 |
| `ACCOMMODATION_COORDINATES_MISSING` | 422 | 기준 숙소 좌표가 없음 |
| `CHAT_UNAVAILABLE` | 503 | AI API 설정 누락, 타임아웃 또는 일시적 실패 |
| `INTERNAL_SERVER_ERROR` | 500 | 처리되지 않은 서버 오류 |

| 상황 | 화면 동작 |
|---|---|
| 숙소를 찾을 수 없음 | 숙소를 다시 선택하도록 안내 |
| 입력값 검증 실패 | 입력창 주변에 검증 메시지 표시 |
| 응답 생성 실패 | 기존 대화를 유지하고 다시 시도 제공 |
| 네트워크 오류 | 연결 실패 안내와 다시 시도 제공 |

AI API가 실패했을 때 정상 응답 안에 오류 문장을 넣지 않는다. 프런트가 실패와 정상 답변을 구분하고 재시도를 제공할 수 있도록 반드시 `503` 오류로 응답한다.

### 3.6 백엔드 추천 처리 규칙

`POST /api/chat`은 다음 순서로 처리한다.

1. `accommodation_id`로 공통 장소 서비스의 장소를 조회한다.
2. 장소가 `숙박` 카테고리인지, 위도·경도가 있는지 검증한다.
3. 질문과 `history`에서 요청 카테고리와 제외 조건을 해석한다.
4. 공통 장소 데이터에서 좌표가 있는 후보를 필터링한다.
5. 숙소와 후보 사이의 Haversine 직선거리를 서버에서 계산한다.
6. 거리순으로 정렬한 뒤 중복을 제거하고 최대 5개를 선택한다.
7. 선택된 실제 장소 데이터만 LLM 프롬프트에 제공한다.
8. 응답의 장소 배열은 LLM 결과가 아니라 6단계의 서버 데이터로 구성한다.

질문에서 카테고리를 특정하지 않으면 `숙박`을 제외한 지원 카테고리 전체를 후보로 사용한다. `문화시설`, `관광지`, `레포츠`, `쇼핑`, `여행코스`, `축제공연행사` 등 장소 API의 정규화된 카테고리명을 그대로 사용한다.

### 3.7 현재 백엔드 구현에서 필요한 수정

현재 `localhub-back`의 `/api/chat`은 존재하지만 다음 수정이 필요하다.

| 위치 | 현재 문제 | 필요한 수정 |
|---|---|---|
| `schemas/chat.py` | `accommodation_id: int` | 실제 장소 ID 형식에 맞게 `str`로 변경 |
| `schemas/chat.py` | `history`가 임의의 `dict` 배열이며 가변 기본값 사용 | 역할·길이를 검증하는 모델과 `default_factory=list` 사용 |
| `api/chat.py` | 요청의 `history`를 서비스에 전달하지 않음 | `chat(..., history=request.history)` 전달 |
| `chat_service.py` | 공통 장소 서비스가 아닌 별도 JSON repository 사용 | `place_service`/`data_loader`의 정규화된 장소 재사용 |
| `recommendation_service.py` | repository는 `name`을 만들지만 `title`을 읽음 | 공통 `PlaceRecord.name` 사용으로 통일 |
| 추천 장소 ID | 원본 `contentid`만 반환해 장소 API ID와 불일치 | 공통 장소 API의 접두사가 포함된 `id` 반환 |
| `PlaceResult` | 주소·이미지·추천 이유가 없음 | 3.4의 구조화된 필드 추가 |
| AI 오류 처리 | 오류도 정상 답변 문자열로 반환 | `CHAT_UNAVAILABLE` 503으로 변환 |

새로운 챗봇 엔드포인트를 하나 더 만들지 않고 기존 `POST /api/chat`의 계약을 위 명세로 교체한다.

## 4. MVP 제외 범위

다음 기능은 유용하지만 이번 챗봇 MVP의 필수 완료 조건에는 포함하지 않는다.

- 추천 장소로 여행 일정을 자동 생성하거나 게시글 작성 화면에 전달
- 날씨, 실시간 교통, 대중교통 소요 시간 연동
- 실제 도로 기반 거리와 최적 경로 계산
- 로그인 기반 사용자 개인화
- 좋아요·싫어요 및 추천 품질 학습
- 새로고침 이후 대화 기록 복구
- 여러 숙소를 동시에 기준으로 추천
- 음성 입력과 음성 답변
- 커뮤니티 게시글 검색
- 챗봇을 모든 페이지에 표시하는 전역 플로팅 위젯

여행 일정 연동과 게시글 검색은 장소 추천 흐름이 안정된 다음 단계에서 추가한다.

## 5. 프런트 상태 모델 최소안

```ts
interface ChatMessage {
  id: string
  role: 'user' | 'assistant'
  content: string
  places?: RecommendedPlace[]
  suggestedQuestions?: string[]
  status?: 'sending' | 'success' | 'error'
}

interface RecommendedPlace {
  id: string
  name: string
  category: string
  address: string | null
  latitude: number
  longitude: number
  image_url: string | null
  distance_km: number
  reason: string
}
```

페이지가 관리해야 하는 핵심 상태는 다음으로 제한한다.

- `selectedAccommodation`
- `messages`
- `selectedPlaceId`
- `requestState`

## 6. 인수 테스트

- [ ] 실제 장소 API에서 숙소를 검색하고 선택할 수 있다.
- [ ] 숙소 미선택 상태에서는 추천 질문이 전송되지 않는다.
- [ ] 질문을 보내면 실제 `/api/chat` 응답이 대화창에 표시된다.
- [ ] 응답의 장소는 모두 장소 DB의 유효한 `id`를 가진다.
- [ ] 추천 장소마다 이유와 직선거리가 표시된다.
- [ ] 추천 카드와 지도 마커 선택 상태가 양방향으로 동기화된다.
- [ ] 후속 질문이 같은 숙소와 이전 답변의 맥락을 사용한다.
- [ ] 숙소 변경 시 기존 대화·추천·세션이 초기화된다.
- [ ] 요청 실패 후 기존 대화를 잃지 않고 다시 시도할 수 있다.
- [ ] 데스크톱과 모바일에서 메시지 입력창과 지도를 쉽게 찾을 수 있다.
- [ ] 데이터에 없는 장소·거리·날짜를 답변에 생성하지 않는다.

## 7. 기존 기획 문서와의 관계

`site_map_plan.md`의 C-01은 전역 챗봇 위젯과 게시글 검색까지 포함하고 있지만, 현재 구현은 `/chatbot` 독립 페이지의 숙소 기반 여행 추천 기능이다. 이 문서는 현재 구현을 완성하는 MVP 범위를 기준으로 한다.

전역 위젯과 게시글 검색을 필수 요구사항으로 유지해야 한다면 본 문서의 제외 범위에서 제거하고 별도 구현 일정을 잡아야 한다. 두 형태를 동시에 구현하기 전에 제품 범위를 먼저 확정한다.
