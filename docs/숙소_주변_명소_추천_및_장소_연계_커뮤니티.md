지금 추가로 얘기 나온 아이디어 중에 숙소 위치 기반으로 주변 명소들을 추천해주는 기능을 넣자는데 어디 파트에 넣어야 될까? 개인적으로 지금 게시판 기능이 단순 게시판 기능밖에 없고 관광명소 데이터랑 연계가 좀 부족한 것 같아서, 그쪽에 뭐 좀 추가하고 싶어.

---

이 기능은 **게시판 추가 기능으로 넣기보다, ‘지도·장소 탐색’ 영역의 Should-have 기능**으로 두는 게 자연스럽습니다. 다만 게시판과 관광 데이터의 연결이 부족하다는 문제는 맞으므로, **게시글에 관련 장소를 연결하는 최소 기능**을 함께 넣는 구성이 좋습니다.

제공 데이터에는 숙박을 구분하는 `contenttypeid=32`와 모든 장소의 `mapx`, `mapy` 좌표가 있으므로, 외부 지도·경로 API 없이도 숙소와 가까운 명소를 거리순으로 계산할 수 있습니다.

## 화면에서는 이렇게 넣는 게 좋습니다

가장 자연스러운 사용자 흐름은 다음입니다.

```text
지도 화면
→ 숙박 필터 선택
→ 숙소 마커 클릭
→ 숙소 정보 패널 표시
→ 주변 명소 5개 표시
→ [이 숙소 관련 글쓰기]
```

게시글을 작성하면 상세 화면은 다음처럼 구성합니다.

```text
게시글 제목
게시글 내용

[관련 숙소]
숙소명 / 주소 / 지도에서 보기

[이 숙소 주변 추천 명소]
명소 1 — 0.8km
명소 2 — 1.4km
명소 3 — 2.1km
```

이렇게 하면 게시판을 별도의 기능으로 놔두지 않고, **숙소 또는 관광지에 대한 후기·질문·정보 공유 공간**으로 보이게 만들 수 있습니다.

## 권장 MVP 범위

3일 일정에서는 범위를 아래까지만 잡는 것이 안전합니다.

### 반드시 구현할 부분

### 시간이 남으면 추가할 부분

1. 반경 1km·3km·5km 선택
2. 관광지·문화시설·축제 등 추천 유형 필터
3. 장소별 관련 게시글 목록
4. 추천 장소를 지도에서 강조 표시

## 데이터 모델 변경

관광 데이터를 SQLite에 전부 저장하지 않는다면, 게시글 테이블에 TourAPI의 `contentid`만 선택적으로 저장하면 됩니다.

```python
class Post(Base):
    __tablename__ = "posts"

    id = Column(Integer, primary_key=True)
    title = Column(String, nullable=False)
    content = Column(Text, nullable=False)
    password = Column(String, nullable=False)

    # 관련 숙소 또는 관광지의 TourAPI contentid
    related_content_id = Column(String, nullable=True)
```

관련 장소의 이름·주소·좌표를 게시글 테이블에 중복 저장할 필요는 없습니다. 게시글을 조회할 때 `related_content_id`를 사용해 로드된 관광 JSON에서 장소를 찾으면 됩니다.

다만 제공 데이터가 나중에 변경될 가능성까지 고려한다면 다음처럼 일부 정보를 스냅샷으로 저장할 수도 있습니다.

```python
related_content_id = Column(String, nullable=True)
related_place_title = Column(String, nullable=True)
```

3일 MVP에서는 `related_content_id` 하나만으로 충분합니다.

## 거리 계산 방식

여기서 말하는 ‘주변’은 **실제 도로 이동 거리**가 아니라 두 좌표 간 직선거리입니다. 화면에도 “직선거리 기준”이라고 표시하는 것이 안전합니다.

`mapx`, `mapy`는 문자열이므로 숫자로 변환한 뒤 Haversine 공식을 사용하면 됩니다. 스키마에도 좌표 사용 시 실수 변환이 필요하다고 명시되어 있습니다. 

```python
from math import asin, cos, radians, sin, sqrt


def haversine_km(
    lat1: float,
    lon1: float,
    lat2: float,
    lon2: float,
) -> float:
    earth_radius_km = 6371.0

    d_lat = radians(lat2 - lat1)
    d_lon = radians(lon2 - lon1)

    a = (
        sin(d_lat / 2) ** 2
        + cos(radians(lat1))
        * cos(radians(lat2))
        * sin(d_lon / 2) ** 2
    )

    return 2 * earth_radius_km * asin(sqrt(a))
```

데이터 건수가 아주 많지 않다면 모든 장소와 거리를 계산한 뒤 정렬해도 충분합니다. 처음부터 공간 인덱스나 복잡한 지리 데이터베이스를 도입할 필요는 없습니다.

## 최종 권장안

기능 이름은 다음처럼 잡는 것이 가장 명확합니다.

> **숙소 주변 명소 추천 및 장소 연계 커뮤니티**

단순히 ‘숙소 주변 추천’만 구현하면 지도 부가기능으로 끝납니다. **숙소에서 게시글 작성 → 게시글에 숙소 표시**까지 연결해야, 현재 느끼는 “커뮤니티와 관광 데이터가 따로 논다”는 문제가 해결됩니다.

다만 일정상 우선순위는 다음 순서를 지키는 것이 좋습니다.

> 게시판 CRUD 완료 → 기본 지도 완료 → 숙소 주변 추천 → 게시글 장소 연결 → 장소별 관련 게시글 목록

마지막 ‘장소별 관련 게시글 목록’은 시간이 부족하면 빼도 전체 흐름에는 문제가 없습니다.
